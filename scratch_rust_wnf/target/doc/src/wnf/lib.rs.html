7//! Safe bindings for the Windows Notification Facility
8//!
9//! The *Windows Notification Facility (WNF)* is a registrationless publisher/subscriber mechanism that was introduced
10//! in Windows 8 and forms an undocumented part of the Windows API.
11//!
12//! This crate provides safe Rust abstractions over (a part of) this API. If you are looking for raw bindings to the
13//! API, take a look at the [`ntapi`](https://docs.rs/ntapi/latest/ntapi/) crate.
14//!
15//! Note that while great care was taken in making these abstractions memory-safe, there cannot be a guarantee due to
16//! the undocumented nature of the API.
17//!
18//! This is a Windows-only crate and will fail to compile on other platforms. If you target multiple platforms, it is
19//! recommended that you declare it as a
20//! [platform specific dependency](https://doc.rust-lang.org/cargo/reference/specifying-dependencies.html#platform-specific-dependencies).
21//!
22//! # How WNF works
23//!
24//! WNF is built upon the core concept of a *state name*. Processes can publish to and subscribe to a state name,
25//! represented by a 64-bit identifier. In this crate, in order to distinguish between such an identifier and the actual
26//! operating system object it represents, we call the identifier a *state name*, while the underlying object will be
27//! referred to as a *state*.
28//!
29//! A state can have different *lifetimes*:
30//! - A *well-known* state is provisioned with the system and cannot be created or deleted.
31//! - A *permanent* state can be created and stays alive even across system reboots until it is explicitly deleted.
32//! - A *persistent* or *volatile* state can be created and stays alive until the next system reboot or until it is
33//!   explicitly deleted.
34//! - A *temporary* state can be created and stays alive until the process it was created from exits or until it is
35//!   explicitly deleted.
36//!
37//! For details, see [`StateLifetime`].
38//!
39//! A state has an associated payload, called the *state data* or *state value*, of up to 4 KB in size. Processes can
40//! query and update these data and subscribe to changes of the data. Furthermore, a state has an associated *change
41//! stamp*, which starts at zero when the state is created and increases by one on every update of the data.
42//!
43//! A state that lives across system reboots (i.e. with *well-known* or *permanent* lifetime) can be configured to
44//! persist its data across reboots. Otherwise, the state itself stays alive but its data is reset on reboots.
45//!
46//! A state can have different *data scopes* that control whether it maintains multiple independent instances of its
47//! data that are scoped in different ways. See [`DataScope`] for the available options.
48//!
49//! A state can have an associated *type id*, which is a GUID that identifies the type of the data. While WNF does not
50//! maintain a registry of types itself, it can ensure that state data are only updated if the correct type id is
51//! provided. This can be useful if you maintain your own type registry or just want to avoid accidental updates with
52//! invalid data.
53//!
54//! Access to a state is secured by a standard
55//! [Windows Security Descriptor](https://learn.microsoft.com/en-us/windows/win32/secauthz/security-descriptors). In
56//! addition, creating a *permanent* or *persistent* state or a state with *process* scope requires the
57//! `SeCreatePermanentPrivilege` privilege.
58//!
59//! The WNF mechanism, though officially undocumented, has been described by various sources. Its API is part of the
60//! Windows Native API exposed through `ntdll.dll` and has been (partly) reverse engineered and described. For details,
61//! refer to these sources:
62//! - [A. Allievi et al.: Windows Internals, Part 2, 7th Edition](https://www.microsoftpressstore.com/store/windows-internals-part-2-9780135462331),
63//!   p. 224ff.
64//! - [Quarkslab's Blog: Playing with the Windows Notification Facility (WNF)](https://blog.quarkslab.com/playing-with-the-windows-notification-facility-wnf.html)
65//! - [A. Ionescu, G. Viala: The Windows Notification Facility: Peeling the Onion of the Most Undocumented Kernel Attack
66//!   Surface Yet](https://www.youtube.com/watch?v=MybmgE95weo), Talk at black hat USA 2018
67//! - [A. Ionescu, G. Viala: WNF Utilities 4 Newbies (WNFUN)](https://github.com/ionescu007/wnfun), including a list of
68//!   the names of well-known states
69//!
70//! # What this crate offers
71//!
72//! This crate provides memory-safe abstractions over most of the WNF API to accomplish these tasks:
73//! - Create and delete a state
74//! - Query information on a state
75//! - Query and update state data
76//! - Subscribe to state data
77//!
78//! Subscribing uses higher-level functions from `ntdll.dll` whose names start with `Rtl`, standing for *runtime
79//! library*:
80//! - `RtlSubscribeWnfStateChangeNotification`
81//! - `RtlUnsubscribeWnfStateChangeNotification`
82//!
83//! The other features use more low-level functions from `ntdll.dll` whose names start with `Nt*`:
84//! - `NtCreateWnfStateName`
85//! - `NtDeleteWnfStateName`
86//! - `NtQueryWnfStateNameInformation`
87//! - `NtQueryWnfStateData`
88//! - `NtUpdateWnfStateData`
89//!
90//! In addition, this crate provides some higher-level abstractions:
91//! - Apply a transformation to state data
92//! - Replace state data
93//! - Wait for updates of state data (in both blocking and async variants)
94//! - Wait until state data satisfy a certain condition (in both blocking and async variants)
95//!
96//! The following WNF features are currently not supported:
97//! - Subscriptions in meta-notification mode, i.e. subscribing to consumers becoming active or inactive or publishers
98//!   terminating
99//! - Event aggregation through the *Common Event Aggregator* to subscribe to updates of one out of multiple states
100//! - Kernel mode
101//!
102//! # Representing states
103//!
104//! There are two types in this crate that represent states: [`OwnedState<T>`] and [`BorrowedState<'_,
105//! T>`](BorrowedState). They work in a similar way as the [`OwnedHandle`](std::os::windows::io::OwnedHandle) and
106//! [`BorrowedHandle<'_>`](std::os::windows::io::BorrowedHandle) types from the standard library:
107//! - An [`OwnedState<T>`] has no lifetime, does not implement [`Copy`] or [`Clone`] and deletes the represented state
108//!   on drop.
109//! - A [`BorrowedState<'_, T>`](BorrowedState) has a lifetime, implements [`Copy`] and [`Clone`] and does *not* delete
110//!   the represented state on drop.
111//!
112//! Note that copying/cloning a [`BorrowedState<'_, T>`](BorrowedState) just copies the borrow, returning another borrow
113//! of the same underlying state, rather than cloning the state itself.
114//!
115//! You can abstract over ownership (i.e. whether a state is owned or borrowed) through the [`AsState`] trait and its
116//! [`as_state`](AsState::as_state) method. This trait is similar to the [`AsHandle`](std::os::windows::io::AsHandle)
117//! trait in the standard library and is implemented by both [`OwnedState<T>`] and [`BorrowedState<'_,
118//! T>`](BorrowedState) as well as any type that derefs to one of these types. Calling [`as_state`](AsState::as_state)
119//! on an [`OwnedState<T>`] borrows it as a [`BorrowedState<'_, T>`](BorrowedState) while calling it on a
120//! [`BorrowedState<'_, T>`](BorrowedState) just copies the borrow.
121//!
122//! You can obtain an instance of [`OwnedState<T>`] or [`BorrowedState<'_, T>`](BorrowedState) in the following ways:
123//! - Create a new owned state through the [`StateCreation::create_owned`] method<br> This is common for temporary
124//!   states, for which there is also the [`OwnedState::create_temporary`] shortcut method.
125//! - Create a new state and statically borrow it through the [`StateCreation::create_static`] method<br> This is common
126//!   for permanent or persistent states.
127//! - Statically borrow an existing state through the [`BorrowedState::from_state_name`] method<br> This is common for
128//!   well-known states.
129//!
130//! An owned state can be "leaked" as a statically borrowed state through the [`OwnedState::leak`] method, while a
131//! borrowed state can be turned into an owned state through the [`BorrowedState::to_owned_state`] method.
132//!
133//! # Representing state data
134//!
135//! The types [`OwnedState<T>`] and [`BorrowedState<'_, T>`](BorrowedState) are generic over a type `T` that describes
136//! the shape of the data associated with the state.
137//!
138//! The state types themselves impose no trait bounds on the data type. However, in order for querying or updating state
139//! data to be memory-safe, the data type needs to satisfy certain conditions:
140//! - Querying state data as a `T` requires that any byte slice whose length is the size of `T` represent a valid `T` or
141//!   that it can at least be checked at runtime whether it represents a valid `T`.
142//! - Updating state data from a `T` requires that the type `T` contain no uninitialized (i.e. padding) bytes.
143//!
144//! These conditions cannot be checked at runtime and hence need to be encoded in the type system.
145//!
146//! Note that querying state data as a `T` also requires that the size of the state data match the size of `T` in the
147//! first place, but this condition can be checked at runtime. In fact, the data type can also be a slice type `[T]`, in
148//! which case the size of the state data is required to be a multiple of the size of `T`.
149//!
150//! Defining how to properly encode the above conditions in the type system is part of the scope of the
151//! [Project "safe transmute"](https://github.com/rust-lang/project-safe-transmute), which is still in
152//! [RFC](https://github.com/jswrenn/project-safe-transmute/blob/rfc/rfcs/0000-safe-transmute.md#safe-transmute-rfc)
153//! stage. However, there are various third-party crates that define (unsafe) traits encoding the above conditions,
154//! among them being the [bytemuck](https://docs.rs/bytemuck/1/bytemuck) and
155//! [zerocopy](https://docs.rs/zerocopy/0/zerocopy) crates. Both of them implement the appropriate traits for many
156//! standard types and also provide macros to derive them for your own types, checking at compile-time whether a type
157//! satisfies the necessary conditions and enabling you to avoid unsafe code in most cases.
158//!
159//! The `wnf` crate does not have a hard dependency on any of these crates. Instead, it defines its own
160//! (unsafe) traits that are modelled after the traits from the [bytemuck](https://docs.rs/bytemuck/1/bytemuck) crate
161//! with the same names:
162//! - [`AnyBitPattern`] and [`CheckedBitPattern`] encode the requirements for querying state data
163//! - [`NoUninit`] encodes the requirements for updating state data
164//!
165//! These traits are already implemented for many standard types. In case your code already makes use of one of the
166//! [bytemuck](https://docs.rs/bytemuck/1/bytemuck) or [zerocopy](https://docs.rs/zerocopy/0/zerocopy) crates or you want
167//! to take advantage of the derive macros provided by either of those crates, you can do the following:
168//! - Enable the `bytemuck_v1` resp. `zerocopy` feature of this crate (producing a dependency on [bytemuck](https://docs.rs/bytemuck/1/bytemuck)
169//!   v1 resp. [zerocopy](https://docs.rs/zerocopy/0/zerocopy))
170//! - Implement the appropriate trait from one of these crates for your type, e.g. by using a derive macro
171//! - Derive the corresponding trait from the `wnf` crate using the [`derive_from_bytemuck_v1`] resp.
172//!   [`derive_from_zerocopy`] macros. See the documentations of these macros for examples.
173//!
174//! All methods that query state data require the data type to
175//! - either be a type `T` that implements [`CheckedBitPattern`]
176//! - or be a slice type `[T]` where the element type `T` implements [`CheckedBitPattern`]
177//!
178//! In addition, they exist in non-boxed and boxed variants, e.g. [`OwnedState::get`] and [`OwnedState::get_boxed`]. The
179//! non-boxed variants query the data by value (i.e. on the stack), but require the data type `T` to be [`Sized`].
180//! The boxed variants work for both sized and unsized data types `T` and query the data as a [`Box<T>`] (i.e. allocated
181//! on the heap). All of this is encapsulated in the [`Read<T>`] trait where
182//! - `T: Read<T>` means that `T` can be read by value
183//! - `T: Read<Box<T>>` means that `T` can be read as a box
184//!
185//! Note that methods that *update* state data don't make a distinction between sized and unsized types/slices or boxed
186//! and non-boxed variants because they take the data by reference.
187//!
188//! If you want to be able to support arbitrary state data without any restriction on the size (apart from the upper
189//! bound of 4 KB), you can always use a byte slice `[u8]` as the data type. In the rare case that you want to query a
190//! state without caring about the content of the data at all, you can use the [`OpaqueData`] type. This is useful e.g.
191//! if you just want to query the size or if you want to check if you have the right permissions to query the state.
192//!
193//! # Examples
194//!
195//! For more detailed examples, see the `examples` folder in the crate repository. Some common use cases:
196//!
197//! ## Creating a state and querying/updating its data
198//!
199//! ```
200//! # fn main() -> Result<(), Box<dyn std::error::Error>> {
201//! use wnf::{CreatableStateLifetime, DataScope, OwnedState, StateCreation};
202//!
203//! let state: OwnedState<u32> = StateCreation::new()
204//!     .lifetime(CreatableStateLifetime::Temporary)
205//!     .scope(DataScope::Machine)
206//!     .create_owned()?;
207//!
208//! state.set(&42)?;
209//!
210//! assert_eq!(state.get()?, 42);
211//! # Ok(()) }
212//! ```
213//!
214//! ## Reading a wide string from a well-known state
215//!
216//! ```
217//! # fn main() -> Result<(), Box<dyn std::error::Error>> {
218//! use std::ffi::{OsStr, OsString};
219//! use std::os::windows::ffi::OsStringExt;
220//!
221//! use wnf::BorrowedState;
222//!
223//! const WNF_SHEL_DESKTOP_APPLICATION_STARTED: u64 = 0x0D83063EA3BE5075;
224//!
225//! let state = BorrowedState::<[u16]>::from_state_name(WNF_SHEL_DESKTOP_APPLICATION_STARTED);
226//!
227//! let last_application_started_wide = {
228//!     let data = state.get_boxed()?;
229//!     OsString::from_wide(&data)
230//! };
231//!
232//! println!("{}", last_application_started_wide.to_string_lossy());
233//! # Ok(()) }
234//! ```
235//!
236//! ## Subscribing to a well-known state
237//!
238//! This requires the `subscribe` feature flag to be enabled.
239//! ```
240//! # #![no_std]
241//! #
242//! # extern crate std as real_std;
243//! #
244//! # mod std {
245//! #     pub(crate) mod io {
246//! #         pub(crate) trait Read {}
247//! #         pub(crate) struct Stdin;
248//! #
249//! #         impl Stdin {
250//! #             pub(crate) fn read_exact(&self, _: &mut [u8]) -> real_std::io::Result<()> {
251//! #                 Ok(())
252//! #             }
253//! #         }
254//! #
255//! #         pub(crate) fn stdin() -> Stdin {
256//! #             Stdin
257//! #         }
258//! #     }
259//! # }
260//! #
261//! # macro_rules! println {
262//! #     ($($arg:tt)*) => { }
263//! # }
264//! #
265//! # fn main() -> Result<(), real_std::boxed::Box<dyn real_std::error::Error>> {
266//! use std::io::{stdin, Read};
267//!
268//! use wnf::{BorrowedState, DataAccessor, OwnedState, SeenChangeStamp};
269//!
270//! const WNF_PO_DISCHARGE_ESTIMATE: u64 = 0x41C6013DA3BC5075;
271//!
272//! let state = BorrowedState::<u64>::from_state_name(WNF_PO_DISCHARGE_ESTIMATE);
273//!
274//! let subscription = state.subscribe(
275//!     |accessor: DataAccessor<'_, _>| {
276//!         println!("Estimated battery duration in seconds: {}", accessor.get().unwrap());
277//!     },
278//!     SeenChangeStamp::Current,
279//! )?;
280//!
281//! stdin().read_exact(&mut [0u8])?; // wait for keypress
282//!
283//! subscription.unsubscribe()?;
284//! # Ok(()) }
285//! ```

287//! # Tracing
288//!
289//! This crate emits diagnostic information using the [`tracing`](https://docs.rs/tracing/latest/tracing) crate whenever there is an interaction with the WNF
290//! API. For guidance on how to subscribe to this information, consult the [`tracing`](https://docs.rs/tracing/latest/tracing) documentation. The general
291//! structure is like this:
292//! - For every invocation of a WNF API function by the `wnf` crate, an [`Event`](https://docs.rs/tracing/latest/tracing/struct.Event.html)
293//!   with the following payload is emitted:
294//!   - The [`target`](https://docs.rs/tracing/latest/tracing/struct.Metadata.html#method.target) is always
295//!     `wnf::ntapi`.
296//!   - The [`level`](https://docs.rs/tracing/latest/tracing/struct.Metadata.html#method.level) is [`DEBUG`](https://docs.rs/tracing/latest/tracing/struct.Level.html#associatedconstant.DEBUG).
297//!   - The message is the name of the WNF API routine.
298//!   - The [`fields`](https://docs.rs/tracing/latest/tracing/struct.Metadata.html#method.fields) consist of three
299//!     groups:
300//!     - `result`: The result (return code) of the invocation
301//!     - `input.*`: Inputs of the invocation, depending on the routine
302//!     - `output.*`: Outputs of the invocation, depending on the routine
303//! - For every invocation of a callback function in the `wnf` crate, a [`Span`](https://docs.rs/tracing/latest/tracing/struct.Span.html)
304//!   with the following payload is emitted:
305//!   - The [`target`](https://docs.rs/tracing/latest/tracing/struct.Metadata.html#method.target) is always
306//!     `wnf::ntapi`.
307//!   - The [`level`](https://docs.rs/tracing/latest/tracing/struct.Metadata.html#method.level) is [`TRACE`](https://docs.rs/tracing/latest/tracing/struct.Level.html#associatedconstant.TRACE).
308//!   - The [`name`](https://docs.rs/tracing/latest/tracing/struct.Metadata.html#method.name) is `WnfUserCallback`.
309//!   - The [`fields`](https://docs.rs/tracing/latest/tracing/struct.Metadata.html#method.fields) are all named
310//!     `input.*` and contain the inputs of the invocation.
311//!
312//! See the `examples` folder in the crate repository for examples on how to subscribe to these events and spans.
313//!
314//! # Cargo features
315//!
316//! This crate has various [feature flags](https://doc.rust-lang.org/cargo/reference/features.html), none of which are
317//! enabled by default. They fall into two groups:
318//!
319//! - Features enabling compatibility with other crates:
320//!   - `bytemuck_v1`: Enables the optional [bytemuck](https://docs.rs/bytemuck/1/bytemuck) dependency and provides the
321//!     [`derive_from_bytemuck_v1`] macro
322//!   - `uuid`: Enables the optional [uuid](https://docs.rs/uuid/1/uuid) dependency and provides conversions between the
323//!     [`uuid::Uuid`](https://docs.rs/uuid/1/uuid/struct.Uuid.html) and [`wnf::GUID`](crate::GUID) types
324//!   - `winapi`: Enables the optional [winapi](https://docs.rs/winapi/latest/winapi) dependency and provides conversions
325//!     between the [`winapi::shared::guiddef::GUID`](https://docs.rs/winapi/latest/winapi/shared/guiddef/struct.GUID.html)
326//!     and [`wnf::GUID`](crate::GUID) types
327//!   - `windows`: Provides conversions between the [`windows::core::GUID`](https://docs.rs/windows/latest/windows/core/struct.GUID.html)
328//!     and [`wnf::GUID`](crate::GUID) types (the `windows` dependency is not optional because it is also used by `wnf`
329//!     internally)
330//!   - `windows_permissions`: Enables the optional [windows-permissions](https://docs.rs/windows-permissions/latest/windows_permissions)
331//!     dependency and enables the use of [`windows_permissions::SecurityDescriptor`](https://docs.rs/windows-permissions/latest/windows_permissions/struct.SecurityDescriptor.html)
332//!     when creating a state
333//!   - `zerocopy`: Enables the optional [zerocopy](https://docs.rs/zerocopy/0/zerocopy) dependency and provides the
334//!     [`derive_from_zerocopy`] macro
335//!
336//! - Features enabling functionality that uses the higher-level `Rtl*` functions from `ntdll.dll` (see above):
337//!   - `subscribe`: Enables subscribing to state updates
338//!   - `wait_blocking`: Enables blocking waits for state updates, implies the `subscribe` feature
339//!   - `wait_async`: Enables async waits for state updates, implies the `subscribe` feature
340//!
